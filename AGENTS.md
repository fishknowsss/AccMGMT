# AccMGMT 项目指引

## 沟通

- 非代码内容默认使用中文。
- 代码、命令、字段名、错误栈保留原始语言。

## 产品定位

AccMGMT 是工作室内部的 Runway 账号预约与占用看板。

- 所有账号默认是 Runway 95 美元 Unlimited 计划。
- 网站不存密码。
- 网站只展示账号邮箱、编号、状态、使用人、小组、项目、时间段、下一预约、续费日期。
- 核心页面是资源占用看板，不是通用后台。

## 信息架构

页面只允许以下主视图：

- `Runway 看板`：查看账号状态、筛选账号、立即使用、预约、结束使用。
- `成员小组`：查看小组成员和占用概览。
- `账号设置`：站点设置和账号信息维护。

禁止新增：

- 账号详情页
- 复杂日历页
- 使用记录独立页面
- 多平台管理页
- 额度/用量页
- 密码管理字段或密码展示

## UI 规则

- Do not render implementation notes in the UI.
- 不要在 UI 中出现实现说明、调试说明、隐藏入口说明、mock、本地环境、脚本名、命令名、内部技术名。
- 不要把“上上下下左右左右BA”等开发入口提示渲染给普通用户。
- 中文文案自然直接，面向内部中文用户。
- 按钮优先 2 到 4 个字，例如：`使用`、`预约`、`结束`、`保存`、`取消`。
- 标签少而准，不堆术语。
- 空状态只说用户下一步该做什么。
- 避免 tooltip 式废话和细小灰色解释文案。
- 优先用留白、对齐、字号、层级表达结构，不靠堆边框和阴影。

## 视觉方向

- 目标气质：Linear / Airtable Interfaces / Resource Guru 式内部资源调度工具。
- 桌面端优先，高密度、清爽、可扫读。
- 表格是核心对象，统计和筛选服务于表格。
- 背景使用浅色中性底，不使用营销页式大 hero、渐变球、装饰插画。
- 状态色低饱和：
  - 空闲：muted green
  - 使用中：muted blue
  - 续费提醒：muted amber
  - 需确认：muted red
- 表格行高约 60 到 68px。
- 数字和时间使用 tabular numbers。
- 避免整页白卡片堆叠成普通 admin template。

## 操作权限

普通成员必须能正常：

- 查看账号状态
- 搜索和筛选
- 使用空闲账号
- 预约账号
- 结束自己的使用

管理员可以结束任何人的使用。

开发者模式只限制账号信息编辑，不限制普通成员的使用、预约、结束等日常操作。

## 开发者模式

- 开发者模式只能在 `账号设置` 视图中通过隐藏按键序列开启。
- 开启前，普通 UI 不显示入口说明。
- 开启后可以显示 `编辑模式` 状态。
- 开启后只允许出现账号信息编辑能力：
  - 新增账号
  - 编辑账号邮箱
  - 编辑账号编号
  - 编辑续费日期
  - 启用或停用账号
- 开发者模式不得引入密码、多平台、额度、账号详情页。

## 数据与状态

- 不要在账号数据里保存 `current_status`。
- 当前状态必须根据当前时间和 confirmed bookings 实时计算。
- 当前时间在某条 confirmed booking 的 `start_time` 和 `end_time` 之间时为使用中。
- 当前没有覆盖当前时间的 booking 时为空闲。
- 下一预约从当前时间之后的 confirmed booking 中取最近一条。
- `end_time` 过去后自动视为空闲，不需要手动释放。
- 同一账号同一时间不能有重叠 booking。
- 冲突条件：`new_start < existing_end AND new_end > existing_start`。

## 组件边界

优先保持这些组件职责：

- `AccountBoardPage`：页面编排和主视图切换。
- `OperationsStrip`：看板统计和占用概览。
- `AccountFilters`：搜索和筛选。
- `AccountTable`：账号资源矩阵。
- `UseNowDialog`：立即使用。
- `BookingDialog`：预约。
- `useAccountsViewModel`：mock 数据、筛选、操作状态、账号编辑。
- `runway-board.ts`：纯状态计算、冲突检测、格式化和账号辅助函数。
- `developer-mode.ts`：隐藏入口序列识别。

不要把数据规则、权限规则和视觉组件揉进一个大文件。

## 代码规范

- 使用 React + TypeScript + Vite。
- 当前样式使用 Tailwind CSS 和少量全局 CSS。
- 保持编辑范围聚焦，不做无关重构。
- 不要添加超出请求范围的功能、注释或抽象。
- 新行为优先补测试，尤其是状态计算、冲突检测、权限边界、开发者模式。

## 交付与版本管理

- 对明确要求实现、修改或修复的任务，在改动完成并通过必要验证后，默认将本次范围内的改动提交到 Git，不保留为未提交状态。
- 提交前检查 diff，只纳入本次任务相关文件，不混入用户已有或其他任务的改动。
- 提交信息使用简洁、准确的中文，说明本次改动目的。
- 推送、发布和部署仅在当前请求、项目流程或用户明确要求时执行，不因默认提交规则自动扩大范围。

## 每次修改后的自查

修改完成后必须运行：

```bash
npm test
npm run lint
npm run build
```

并检查：

- UI 中没有实现说明。
- 没有密码字段。
- 没有多平台字段。
- 没有额度字段。
- 没有账号详情页。
- 没有复杂日历。
- 普通成员仍可使用、预约、结束自己的使用。
- 开发者模式只影响账号信息编辑。
